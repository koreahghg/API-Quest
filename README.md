# API Quest 🗺️

> 초보 개발자를 위한 인터랙티브 API 학습 플랫폼 — **틀려야 배운다**

---

## 프로젝트 한 줄 정의

API Quest는 HTTP 요청의 **실패 경험**을 학습 기회로 전환하는 플랫폼입니다.  
Postman처럼 API를 호출하는 도구가 아닌, 왜 실패했는지·어떻게 고치는지를 스스로 발견하게 만드는 학습 환경입니다.

---

## 타겟 사용자

| 분류 | 설명 |
|------|------|
| **주요 타겟** | 코딩 입문 3~12개월차, 처음 API를 연동해야 하는 개발자 |
| **페르소나 A** | 부트캠프 수료 후 첫 프로젝트에서 백엔드 API를 연동해야 하는 신입 프론트엔드 개발자 |
| **페르소나 B** | HTTP를 이론으로만 배운 CS 1~2학년 학생 |
| **페르소나 C** | Postman을 쓰지만 401·CORS가 뜨면 패닉 상태가 되는 독학 개발자 |
| **제외 대상** | REST API 패턴을 이미 숙지한 중급 이상 개발자 |

---

## 해결하려는 문제

1. **도구는 있지만 이해가 없다**  
   Postman으로 API를 호출할 수 있지만, 401이 왜 뜨는지·CORS가 무엇인지 모른 채로 복붙한다.

2. **에러 메시지가 두렵다**  
   빨간 에러를 마주하면 그냥 닫아버린다. 에러를 읽고 해석하는 경험이 없다.

3. **컨텍스트 없는 문서**  
   공식 API 문서는 "무엇을 하는지"는 설명하지만, "왜 이렇게 설계됐는지"는 알려주지 않는다.

4. **이론과 실습의 단절**  
   HTTP 상태코드를 암기했지만, 실제 요청에서 마주쳤을 때 연결이 안 된다.

## 핵심 가치 제안

> **"틀리는 것이 학습이다"**

API Quest는 잘못된 요청을 보냈을 때 단순히 에러를 보여주는 게 아니라,  
**왜 실패했는지 설명하고 다음 시도를 유도**합니다.  
실패 → 이해 → 재시도의 사이클이 곧 학습입니다.

| 기존 도구 (Postman / Swagger) | API Quest |
|---|---|
| API 호출 도구 | API 학습 플랫폼 |
| 에러 코드만 표시 | 에러 원인 + 해결 방향 안내 |
| 자유 탐색 | 퀘스트 기반 단계적 학습 |
| 결과 중심 | 과정 + 이해 중심 |

## MVP 기능 (v1.0)

### 1. API 요청 플레이그라운드
- HTTP 메서드 선택 (GET / POST / PUT / DELETE / PATCH)
- URL 직접 입력 (Enter 키로 즉시 전송)
- 요청 헤더 편집기 (key-value 추가 / 삭제 / 활성화 토글)
- 요청 바디 편집기 (JSON 포맷 지원, GET/DELETE 시 비활성화)
- 응답 뷰어 (상태코드, 응답 헤더, 바디, 응답시간)

### 2. 상태코드 학습 힌트
- 응답 상태코드에 따른 **한국어 설명 자동 표시**
- "이게 왜 실패했는가?" 컨텍스트 제공
- 흔한 실수 패턴 안내 (예: Authorization 헤더 누락 → 401)

### 3. 퀘스트 시스템 _(다음 PR)_
- 단계별 미션: GET 기본 → 쿼리 파라미터 → 인증 → POST → 에러 핸들링
- 미션별 성공 조건 자동 검증
- 단계적 힌트 공개

## 확장 기능 (v2.0+)

- **퀘스트 채점 시스템** — 정확한 요청 구조 서버 검증
- **오답 노트** — 틀린 요청 히스토리 저장 및 복습
- **API 시나리오 모드** — 실제 서비스(로그인·장바구니)를 시뮬레이션
- **HTTP 개념 인라인 위키** — 각 UI 요소와 연결된 개념 설명 팝업
- **요청 공유** — 내 설정을 링크로 공유
- **환경 변수** — `{{BASE_URL}}`, `{{TOKEN}}` 같은 변수 지원

## 사용자 시나리오

### 시나리오 1 — 첫 인증 API (Sarah의 이야기)
Sarah는 부트캠프를 수료하고 날씨 API를 연동해야 한다.  
API Quest에서 URL을 입력하고 Send를 누르자 **401**이 뜬다.  
힌트 탭을 열면 *"Authorization 헤더가 없거나 잘못된 형식입니다. Bearer 토큰을 확인해보세요."* 메시지가 뜬다.  
Headers 탭에서 `Authorization: Bearer <token>`을 추가하자 200 응답이 온다.  
처음으로 인증 흐름을 "공식 문서가 아닌 직접 경험으로" 이해했다.

### 시나리오 2 — CORS 이해하기 (민준의 이야기)
민준은 React 앱에서 외부 API를 직접 호출하다 CORS 에러를 만났다.  
API Quest 플레이그라운드에서 같은 요청을 보내보고 힌트를 읽는다.  
*"브라우저는 다른 출처의 리소스를 자동으로 차단합니다. 이것이 CORS입니다."*  
왜 프록시 서버가 필요한지, 처음으로 납득한다.
